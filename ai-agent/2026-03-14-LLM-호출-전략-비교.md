# LLM 호출 전략 비교: Stateless vs Stateful 에이전트

### Context
LLM 을 활용한 애플리케이션을 구축할 때, 작업의 성격에 따라 최적의 호출 전략이 달라집니다. 블로그 포스팅 자동화나 뉴스 요약처럼 **'주제 입력 → 본문 생성 → 저장'**으로 끝나는 단발성(Stateless) 작업과, 텔레그램 챗봇처럼 **'대화의 맥락(Context)'**이 중요한 지속적인 대화 작업은 서로 다른 아키텍처가 필요합니다. 사용자의 입력은 이 두 가지 시나리오를 명확히 구분하여 효율적인 LLM 호출 전략을 제시하고 있습니다.

### Core
작업의 성격에 따라 다음과 같이 아키텍처를 분리하여 설계해야 합니다.

**1. 단발성 작업 (Stateless) - HTTP Request 방식**
*   **적합한 시나리오**: 뉴스 요약, 이미지 생성, 일회성 데이터 변환 등.
*   **특징**: 요청과 응답이 독립적이며, 이전 대화 기록을 유지할 필요가 없습니다.
*   **장점**: 가볍고 빠르며, 서버 부하가 적고 확장성이 뛰어납니다.
*   **구현 패턴**:
    ```python
    # 간단한 HTTP Request 예시 (단발성)
    import requests

    def generate_blog_post(topic):
        payload = {
            "model": "claude-3-sonnet",
            "messages": [
                {"role": "user", "content": f"'{topic}'에 대한 블로그 포스팅을 작성해줘."}
            ],
            "max_tokens": 1000
        }
        response = requests.post("https://api.anthropic.com/v1/messages", json=payload)
        return response.json()['content'][0]['text']
    ```

**2. 대화형 작업 (Stateful) - AI Agent 방식**
*   **적합한 시나리오**: 고객 지원 챗봇, 개인 비서, 복잡한 문제 해결 등 맥락이 중요한 작업.
*   **특징**: 메모리(Memory) 관리, 도구 호출(Tool Calling), 페르소나 유지가 필수적입니다.
*   **장점**: 대화 흐름을 유지하며, 필요 시 외부 도구(구글 검색 등) 를 스스로 호출하여 정보를 보완할 수 있습니다.
*   **구현 패턴 (Agent Framework 예시)**:
    ```python
    # 에이전트 프레임워크 예시 (LangChain 또는 유사 라이브러리)
    from langchain.agents import initialize_agent, Tool
    from langchain.memory import ConversationBufferMemory

    # 메모리 설정 (단기/장기 기억 관리)
    memory = ConversationBufferMemory(memory_key="chat_history")

    # 도구 정의 (예: 구글 검색)
    search_tool = Tool(
        name="Google Search",
        func=google_search_function,
        description="실시간 정보를 검색할 때 사용"
    )

    # 에이전트 초기화
    agent = initialize_agent(
        tools=[search_tool],
        llm=llm,
        agent="zero-shot-react-description",
        memory=memory,
        verbose=True
    )

    # 실행
    agent.run("오늘의 뉴스 요약해줘. 그리고 관련 배경지식도 알려줘.")
    ```

### Insight
단발성 작업에서는 HTTP Request 를 통한 직접 호출이 리소스 효율성과 속도 면에서 압도적으로 유리합니다. 반면, 대화형 에이전트는 **기억력(Memory)** 시스템(예: Redis 기반의 단기 기억 또는 벡터 DB 기반의 장기 기억) 과 **도구 호출(Tool Calling)** 능력을 결합해야만 진정한 지능형 상호작용이 가능합니다. 특히 페르소나 설정이 명확한 에이전트는 시스템 프롬프트를 통해 일관된 톤앤매너를 유지하며, 필요 시 외부 API 를 스스로 호출하여 정보의 정확성을 높일 수 있습니다. 따라서 애플리케이션의 목적에 따라 Stateless 와 Stateful 아키텍처를 명확히 분리하여 설계하는 것이 핵심입니다.
